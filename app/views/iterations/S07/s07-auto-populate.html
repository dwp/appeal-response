Auto populate

Hypothesis / Experiment

  • We have seen that
    AWs spend a lot of time populating some parts of the response
    There can be errors in some parts of the form
  • We think this is because AWs have to refer to and copy from various DWP systems to populate parts of the response other than the response reasoning itself
  • So if we Automatically pre-populate some of the form, where we already hold the relevant data in DWP systems.
  • We will see
    AWs will be able to work more efficiently
    Reduced errors or inconsistencies in response documents.
  • We'll know this is true when
    Appeals processing throughout increases with no reduction in quality. 
    Internal quality metrics show accuracy improvements.
    AWs report that they are able to spend more of their time on high-value response writing activity. 

What's new or changed?

  • Automatically populate Appellant details, Representative details, Appeal details, Schedule of evidents and Illness and disabilities from various systems.
  • Clearly show the source of data.

What did we learn?

  • Participants recognised the concept and its value.
  • Emerging finding that auto-populated data would always be checked, and facility to edit data is important.
  • Both participants highlighted challenges in achieving this autopopulation.
  • Both P’s showed recognition of SSC1-sourced data and this seemed to be perceived as trustworthy.
  • Unclear whether the messaging about source of data was recognised by participants immediately.
  • Both P’s think that, even if data still needs to be checked and verified, some autopopulation will speed things up.
  • Time savings of 15 minutes and 40 minutes per case referenced.
  • “At the moment, where UC build is, there’s no way it could do that.”
  • “I’d trust everything up to the schedule of events. I’d still want to check that out. “
